# SAM_GAME - 삼국지 전략 시뮬레이션 (Unity C#)

삼국지 14 스타일의 턴제 전략 시뮬레이션 게임

## 개요

| 항목 | 내용 |
|------|------|
| **엔진** | Unity (C#) |
| **장르** | 턴제 전략 시뮬레이션 |
| **레퍼런스** | 삼국지 14 (코에이) |
| **핵심 메커니즘** | 헥스 기반 영토 칠하기, 군대 이동 시 세력 색 전파 |
| **시대 배경** | 후한 말기 ~ 삼국시대 (190년~) |
| **맵 규모** | 200x150 헥스 그리드 (약 30,000 셀) |
| **도시 수** | 46개 (역사적 거점) |

## 시스템 아키텍처

### 데이터 흐름

```
GameManager (게임 상태/턴 관리)
  │
  ├─ EventManager (이벤트 버스 - 시스템 간 통신)
  │
  ├─ MapInitializer (게임 시작 시 맵 초기화)
  │   ├─ HexGrid + TerrainDataProvider → 30,000 헥스 + 지형 배치
  │   ├─ ProvinceDataProvider → 46개 도시 배치
  │   ├─ InfluenceCalculator → 도시별 영향권 계산 (다중 다익스트라)
  │   ├─ TerritoryManager → 초기 세력 영토 칠하기
  │   ├─ HexChunk → 청크 메시 생성
  │   └─ ProvinceView → 도시 마커 배치
  │
  ├─ MapManager (플레이어 입력)
  │   ├─ 카메라: 줌, 드래그, WASD, 가장자리 이동
  │   └─ 선택: 도시 클릭 → ProvinceSelected, 헥스 클릭 → HexSelected
  │
  ├─ ArmyMovementManager (군대 이동)
  │   ├─ HexPathfinder → A* 경로 탐색
  │   └─ TerritoryManager.PaintHex() → 이동 경로 영토 칠하기
  │
  ├─ BattleManager (전투)
  ├─ ProvinceManager (내정/자원)
  ├─ OfficerManager (무장 관리)
  ├─ DiplomacyManager (외교)
  ├─ AIController (AI 턴 처리)
  │
  └─ UIManager → 정보 패널 표시
```

### 디자인 패턴

| 패턴 | 적용 |
|------|------|
| **싱글턴** | GameManager, UIManager |
| **이벤트 버스** | EventManager (문자열 키 기반 pub/sub) |

## 핵심 시스템 상세

### 1. 헥스 그리드 시스템

- **좌표계**: 큐브 좌표 (q, r, s), Flat-top 레이아웃
- **크기**: 200x150 (오프셋 좌표 기준)
- **셀 데이터**: 순수 C# 클래스 (MonoBehaviour X → 성능)
- **좌표 변환**: 오프셋 ↔ 큐브 ↔ 월드 좌표

### 2. 영토 칠하기 (Territory Painting) — 핵심 메커니즘

삼국지 14의 가장 특징적인 시스템:

1. **초기화**: 다중 소스 다익스트라로 각 도시의 영향권 계산 (지형 비용 반영 → 산/강이 자연 국경)
2. **군대 이동**: 군대가 헥스를 지나가면 소속 세력 색으로 칠함
3. **도시 점령**: 도시 점령 시 해당 도시 영향권 전체가 새 세력으로 전환

### 3. 경로 탐색 (A*)

- 병종별 지형 이동 비용 차별화 (기병은 산 진입 불가, 수군만 바다 이동)
- 휴리스틱: 헥스 거리 × 최소 이동 비용

### 4. 지형 시스템

| 지형 | 보병 | 기병 | 궁병 | 수군 | 방어 보너스 |
|------|------|------|------|------|------------|
| 평지 | 1.0 | 1.0 | 1.0 | ∞ | 1.0x |
| 숲 | 1.5 | 2.0 | 1.5 | ∞ | 1.2x |
| 산 | 3.0 | ∞ | 3.0 | ∞ | 1.5x |
| 강 | 2.0 | 2.0 | 2.0 | 1.0 | 1.3x |
| 관문 | 1.5 | 1.5 | 1.5 | ∞ | 1.8x |
| 성 | 1.0 | 1.0 | 1.0 | ∞ | 2.0x |
| 바다 | ∞ | ∞ | ∞ | 1.0 | 1.0x |

### 5. 렌더링 최적화

- 30,000 헥스를 개별 GameObject로 만들면 성능 파괴
- 20x20 헥스 단위 **청크 메시**로 결합 렌더링
- 영토 변경 시 해당 청크만 메시 재생성
- 버텍스 컬러: 세력 색 70% + 지형 틴트 30% 블렌딩

## 기술 결정 사항

| 결정 | 이유 |
|------|------|
| 큐브 좌표계 | 오프셋 좌표보다 거리/이웃 계산이 단순 |
| HexCell을 순수 클래스로 | 30,000개를 MonoBehaviour로 만들면 성능 문제 |
| 청크 메시 렌더링 | 개별 GO 대신 결합 메시 → 드로우콜 최소화 |
| 다중 소스 다익스트라 | 단순 최근접보다 지형 비용 반영한 자연스러운 국경 |
| 이벤트 버스 (문자열 키) | 시스템 간 느슨한 결합, 확장 용이 |
| JSON 세이브 | 초기 개발 단계에서 디버깅 용이 |
| 절차적 지형 생성 | 텍스처 없이도 개발 가능, 나중에 텍스처로 교체 가능 |
